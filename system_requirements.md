# 家族图谱应用系统需求文档（system_requirements）

## 1. 系统目标与边界
- 面向单编辑者维护、亲友只读访问的家族图谱 Web 系统。
- 第一阶段无需账号登录；通过受控只读链接进行分享。
- 支持家族图谱编辑、可视化浏览、导出与分发。

## 2. 总体架构要求
- 架构形态：前后端分离。
- 前端：
- 单页 Web 应用，负责图谱绘制、交互编辑、只读浏览与导出触发。
- 后端：
- 提供成员/关系/分享配置等 API。
- 负责数据持久化、只读链接校验、导出任务（如采用服务端导出）。
- 存储：
- 结构化数据存储成员与关系。
- 文件存储保存头像与导出产物（可选）。

## 3. 功能模块与系统职责

### 3.1 成员管理模块
- 提供成员 CRUD 能力。
- 支持头像上传与访问 URL 维护。
- 字段级校验与基础数据清洗（空白字符、非法年份、重复姓名提示）。
- 图片处理约束：限制格式（JPG/PNG/WebP）与大小（建议 <= 5MB），生成可展示缩略图。

### 3.2 关系管理模块
- 提供关系创建、更新、删除。
- 关系类型：`parent`、`child`、`spouse`、`sibling`。
- 维护关系一致性约束：
- 禁止自循环关系。
- 禁止明显冲突关系（同一对成员被定义为不相容关系）。

### 3.3 图谱渲染模块
- 支持自动布局（树状代际分层，祖辈在上/小辈在下）+ 手动微调保存。
- 支持平移、缩放、节点聚焦、搜索定位与高亮。
- 中等规模（300-500 节点）下交互保持可用与可读。
- 节点渲染需支持头像显示与兜底占位图。

### 3.4 只读分享模块
- 生成不可预测分享 Token。
- 通过 Token 访问只读页面，不提供写接口入口。
- 支持“启用/停用”分享状态切换。

### 3.5 导出模块
- 输出格式：PNG、PDF。
- 输出模式：当前视图、全图自适应。
- 导出文件需保证文字清晰、线条连续、布局稳定。

## 4. 数据模型要求（逻辑）

### 4.1 Member（成员）
- `id`: 唯一标识
- `name`: 姓名
- `gender`: 性别（可空）
- `birth_year`: 出生年（可空）
- `generation`: 代际标识（可空）
- `avatar_url`: 头像地址（可空）
- `title`: 常用称呼（可空）
- `bio`: 简介（可空）
- `origin`: 籍贯（可空）
- `note`: 备注（可空）
- `is_alive`: 在世状态（可空）
- `created_at` / `updated_at`

### 4.2 Relationship（关系）
- `id`: 唯一标识
- `from_member_id`
- `to_member_id`
- `relation_type`: `parent|child|spouse|sibling`
- `created_at` / `updated_at`

### 4.3 ShareLink（分享）
- `id`: 唯一标识
- `token`: 唯一访问令牌
- `is_enabled`: 是否启用
- `expires_at`: 过期时间（可空）
- `created_at` / `updated_at`

## 5. API 需求（最小集合）
- `GET /api/members`
- `POST /api/members`
- `PUT /api/members/:id`
- `DELETE /api/members/:id`
- `POST /api/members/:id/avatar`（上传头像）
- `GET /api/relationships`
- `POST /api/relationships`
- `DELETE /api/relationships/:id`
- `POST /api/share-links`
- `PATCH /api/share-links/:id`
- `GET /view/:token`（只读入口）
- `POST /api/exports`（触发导出）

## 6. 安全要求
- 只读链接 Token 必须高熵、不可顺序枚举。
- 所有写操作接口默认仅允许编辑端访问（第一阶段可通过部署层白名单或本地内网限制）。
- 文件上传限制类型与大小，防止恶意文件注入。
- API 输入统一做参数校验与错误码规范化。

## 7. 性能与容量要求
- 数据规模：目标支持 500 名成员、1000 条关系。
- 渲染性能：
- 首屏加载（缓存命中）目标 < 3 秒（桌面网络环境）。
- 典型操作（缩放、拖拽、选中）无明显卡顿。
- 头像上传后在界面中反馈时间建议 < 1 秒（本地预览场景）。
- 导出性能：
- 常规图谱导出在可接受时长内完成（目标 < 10 秒，视设备而定）。

## 8. 可靠性与可维护性
- 前后端日志可定位关键错误（导出失败、关系冲突、上传失败）。
- 关键数据操作具备失败回滚或重试机制。
- 数据结构可平滑升级到“多编辑者 + 登录鉴权”模式。

## 9. 可用性与可访问性
- 界面文案清晰，关键按钮语义明确（新增成员、建立关系、导出、分享）。
- 保证基础可访问性：可见焦点、对比度达标、关键功能可键盘操作。
- 移动端至少保障浏览与搜索能力，编辑优先桌面端。

## 10. 技术选型建议（建议项）
- 前端：React + TypeScript + SVG 渲染图谱（便于导出与样式控制）。
- 后端：Node.js（NestJS/Express）+ REST API。
- 数据库：PostgreSQL（关系一致性与查询能力强）。
- 文件存储：S3 兼容对象存储（头像、导出文件）。
- 部署：前后端分离部署，静态站点 + API 服务。
